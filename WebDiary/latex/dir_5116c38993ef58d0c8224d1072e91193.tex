\hypertarget{README.md_autotoc_md57}{}\doxysubsubsection{\texorpdfstring{Terms}{Terms}}\label{README.md_autotoc_md57}

\begin{DoxyItemize}
\item {\bfseries{Full}}\+: builds that contain both the compiler and the runtime.
\item {\bfseries{Compiler}}\+: code that is responsible for compiling template strings into Java\+Script render functions.
\item {\bfseries{Runtime}}\+: code that is responsible for creating Vue instances, rendering and patching virtual DOM, etc. Basically everything minus the compiler.
\item {\bfseries{\href{https://github.com/umdjs/umd}{\texttt{ UMD}}}}\+: UMD builds can be used directly in the browser via a {\ttfamily \texorpdfstring{$<$}{<}script\texorpdfstring{$>$}{>}} tag. The default file from Unpkg CDN at \href{https://unpkg.com/vue}{\texttt{ https\+://unpkg.\+com/vue}} is the Runtime + Compiler UMD build ({\ttfamily vue.\+js}).
\item {\bfseries{\href{http://wiki.commonjs.org/wiki/Modules/1.1}{\texttt{ Common\+JS}}}}\+: Common\+JS builds are intended for use with older bundlers like \href{http://browserify.org/}{\texttt{ browserify}} or \href{https://webpack.github.io}{\texttt{ webpack 1}}. The default file for these bundlers ({\ttfamily pkg.\+main}) is the Runtime only Common\+JS build ({\ttfamily vue.\+runtime.\+common.\+js}).
\item {\bfseries{\href{http://exploringjs.com/es6/ch_modules.html}{\texttt{ ES Module}}}}\+: ES module builds are intended for use with modern bundlers like \href{https://webpack.js.org}{\texttt{ webpack 2}} or \href{http://rollupjs.org/}{\texttt{ rollup}}. The default file for these bundlers ({\ttfamily pkg.\+module}) is the Runtime only ES Module build ({\ttfamily vue.\+runtime.\+esm.\+js}).
\end{DoxyItemize}\hypertarget{README.md_autotoc_md58}{}\doxysubsubsection{\texorpdfstring{Runtime + Compiler vs. Runtime-\/only}{Runtime + Compiler vs. Runtime-\/only}}\label{README.md_autotoc_md58}
If you need to compile templates on the fly (e.\+g. passing a string to the {\ttfamily template} option, or mounting to an element using its in-\/\+DOM HTML as the template), you will need the compiler and thus the full build.

When using {\ttfamily vue-\/loader} or {\ttfamily vueify}, templates inside {\ttfamily \texorpdfstring{$\ast$}{*}.vue} files are compiled into Java\+Script at build time. You don\textquotesingle{}t really need the compiler in the final bundle, and can therefore, use the runtime-\/only build.

Since the runtime-\/only builds are roughly 30\% lighter-\/weight than their full-\/build counterparts, you should use it whenever you can. \hypertarget{README.md_autotoc_md62}{}\doxysubsubsection{\texorpdfstring{Development vs. Production Mode}{Development vs. Production Mode}}\label{README.md_autotoc_md62}
Development/production modes are hard-\/coded for the UMD builds\+: the un-\/minified files are for development, and the minified files are for production.

Common\+JS and ES Module builds are intended for bundlers, therefore we don\textquotesingle{}t provide minified versions for them. You will be responsible for minifying the final bundle yourself.

Common\+JS and ES Module builds also preserve raw checks for {\ttfamily process.\+env.\+NODE\+\_\+\+ENV} to determine the mode they should run in. You should use appropriate bundler configurations to replace these environment variables in order to control which mode Vue will run in. Replacing {\ttfamily process.\+env.\+NODE\+\_\+\+ENV} with string literals also allows minifiers like Uglify\+JS to completely drop the development-\/only code blocks, reducing final file size.\hypertarget{README.md_autotoc_md63}{}\doxysubsubsubsection{\texorpdfstring{Webpack}{Webpack}}\label{README.md_autotoc_md63}
